# Development Estimation Template

This template defines the structure and format for development effort estimation documents based on requirements analysis for the HealthBridge health management platform.

## PATH NAMING CONVENTION

**CRITICAL:** Always use **hyphens** (`-`) not underscores (`_`) in output paths:
- Correct: `reports/requirements-analysis/[TICKET]-dev-estimation.md` (when part of workflow)
- Wrong: `reports/dev_estimation/` (old location, deprecated)

## CRITICAL CONSTRAINTS

| Constraint | Value | Enforcement |
|------------|-------|-------------|
| **Maximum Word Count** | **1500 words** | MANDATORY - Tables only. No prose explanations. |
| **Prerequisites** | **Requirements completeness >= 7/10** | Only generate if requirements are sufficiently complete |
| **Repository Scope** | **Only impacted repos** | Skip repos with no impact |
| **Estimation Level** | **Detailed (task-level)** | Specific hours per file/component modification |
| **Code Examples** | **NONE** | Reference file paths, don't show code |
| **Format** | **Tables > Prose** | Every estimate in a table row |
| **Day Calculation** | **1 developer day = 7 hours** | Use consistently for all day conversions |

### Mandatory vs. Optional Sections

| Section | When to Include |
|---------|----------------|
| **1. Estimation Summary** | **Always** |
| **2. Repository Breakdown** | **Always** (only impacted repos) |
| **3. Cross-Repository Tasks** | Only if multi-repo coordination needed |
| **4. DevOps & Infrastructure** | Only if feature flags, CI/CD, or config changes needed |
| **5. Documentation Tasks** | Only if documentation changes needed |
| **6. Testing & QA Tasks** | Only if dev-side smoke testing or review time is significant |
| **7. Risk & Contingency** | **Always** |
| **8. Total Estimate Summary** | **Always** |
| **9. Confidence & Assumptions** | **Always** (abbreviate to 9.1 + 9.2 if space tight) |
| **10. Parallelization Notes** | Only if multi-developer or multi-repo |

**Space management:** For simple/single-repo features, skip optional sections entirely. For complex multi-repo features, include all applicable sections but abbreviate sub-tables to impacted items only (no "Example: No changes needed" rows).
