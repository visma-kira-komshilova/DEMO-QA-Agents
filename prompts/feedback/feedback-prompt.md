# Feedback Processing Prompt

Analysis logic for the Feedback Agent. For orchestration, see `agents/vscode-chat-participants/feedback.md`. For accuracy report format, see `feedback-template.md`.

---

## Section 10 Parse Rules

Extract each row from the Developer Feedback table:

```markdown
| # | Section | Finding | Verdict | Comment |
```

**Verdict mapping:**

| Developer Input | JSON Value |
|----------------|------------|
| `Valid` | `"valid"` |
| `False Positive` or `FP` | `"false_positive"` |
| `Won't Fix` or `WF` | `"wont_fix"` |
| Empty/blank | Skip row (no feedback given) |

**Overall Accuracy:** Extract number from `___/10`. If blank or non-numeric, save as `null`. Exclude `null` values from average calculations in aggregate mode.

**Additional Comments:** Extract free-text below the feedback table.

---

## Finding Classification

### Section 3.2 Findings (Hotfix Pattern Prevention)

- **Pattern category:** Read directly from the pattern name in the table row (e.g., "Edge Cases (XX%)" → `"Edge Cases"`). Do not infer by keyword matching.
- **Severity:** `fail` → `"warning"`, `warn` → `"suggestion"`.

### Section 6 Findings (Issues Found)

- **Pattern category:** Infer from finding description. If it clearly maps to a hotfix pattern category, use that. Otherwise use `"Other"`.
- **Severity:** Read from subsection: "Critical" → `"critical"`, "Warning" → `"warning"`, "Suggestion" → `"suggestion"`.

## Canonical Feedback JSON Schema

Single source of truth. Used by both Process mode (save) and Aggregate mode (read).

```json
{
  "ticket": "HM-XXXXX",
  "report_file": "<TICKET>-code-review.md",
  "report_date": "YYYY-MM-DD",
  "feedback_date": "YYYY-MM-DD",
  "feedback_mode": "static | interactive",
  "repository": "HealthBridge-Web",
  "risk_level": "Low | Medium | Critical",
  "findings": [
    {
      "id": 1,
      "section": "3.2 | 6",
      "finding": "Description of the finding",
      "pattern_category": "Edge Cases | NULL Handling | Other | ...",
      "severity": "critical | warning | suggestion",
      "verdict": "valid | false_positive | wont_fix",
      "deep_analysis_requested": false,
      "comment": "Developer explanation (optional)"
    }
  ],
  "overall_accuracy": 8,
  "additional_comments": "Free-text developer comments",
  "summary": {
    "total_findings": 6,
    "rated_findings": 6,
    "valid": 4,
    "false_positive": 1,
    "wont_fix": 1,
    "deep_analysis_requested": 0
  }
}
```

**Field notes:**
- `feedback_mode`: `"interactive"` = generated by Code Review Agent's live feedback loop; `"static"` = parsed from filled-in Section 10
- `deep_analysis_requested`: `true` if developer selected "Provide More Information" during interactive mode
- `overall_accuracy`: Developer-provided score (1–10) or `null` if not provided

## Aggregate Metric Calculations

### Verdict Distribution

```
valid_rate     = valid_count / rated_count × 100
fp_rate        = false_positive_count / rated_count × 100
wont_fix_rate  = wont_fix_count / rated_count × 100
```

### Average Accuracy Score

Mean of all `overall_accuracy` values from feedback JSONs. Exclude `null` values.

### Per-Pattern FP Rate

```
pattern_fp_rate = pattern_fp_count / pattern_total_count × 100
```

### High FP Pattern Threshold

Flag pattern category when **both** conditions met:
- ≥ 5 rated findings in that category
- FP rate > 30%

Extract common developer reasons from `comment` fields for suggested prevention rules.

### Trend Analysis (3+ reports required)

- Group by month if data spans multiple months
- Direction: **improving** = accuracy up or FP rate down, **declining** = opposite, **stable** = ±0.5 variation

## False Positive Analysis

Before suggesting new prevention rules:
1. Read `context/code-review-false-positive-prevention.md`
2. Check if a rule already exists for the flagged pattern
3. Only suggest a new rule if no equivalent rule is present

**FP analysis table format:**

| # | Finding | Pattern | Developer Reason | Existing Rule? |
|---|---------|---------|------------------|----------------|
| X | [finding] | [category] | [comment] | Yes (Rule N) / No |

## Pre-Submission Checklists

### Process Mode

- [ ] Report file found and read (not fabricated)
- [ ] Interactive feedback check performed before parsing
- [ ] All verdict mappings correct (Valid/FP/Won't Fix)
- [ ] Pattern categories from Section 3.2 read directly (not inferred)
- [ ] JSON follows canonical schema above
- [ ] Summary counts match findings array
- [ ] Date format is YYYY-MM-DD

### Aggregate Mode

- [ ] All feedback JSONs read from `reports/feedback/`
- [ ] Unparseable files skipped and logged
- [ ] Duplicate tickets deduplicated (most recent by feedback_date)
- [ ] Null accuracy scores excluded from average
- [ ] FP threshold rules applied (≥5 findings AND >30%)
- [ ] Accuracy report follows template structure exactly
- [ ] Existing prevention rules checked before suggesting new ones
